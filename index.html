Що таке ООП?

1. SRP - Single Responsibility Principle

Групувати обов'язки - природньо

Клас повинен мати одну, і лише одна причину для змін

			class Employee{
				calculatePayment();
				createHoursReport();
				writeEmployee();
			}

Обов'язок класу - це вісь змін

Якщо в класу більше одного обов'язку, то обов'язки стають щепленими, взаємозалежними.

Типові приклади порушення SRP

  • God Class
  • Поєднання бізнес-логіки і відображення
  • Поєднання бізнес-логіки і збереження в БД

2. OCP - Open-Closed Principle

Вимоги завжди
змінюються

Програмні одиниці (класи, модулі, функції і тд) повинні бути відкритими для розширень, але закритими до змін

Мати можливість розширювати поведінку модуля без зміни самого модуля 

			function drawAllShapes(shapes){

					case "circle": 
					  drawCirle(shape);
					  break;
					case "square":
					  drawSquare(shape);
					  break;
				  }
				})
			}

Подумаємо про зміни

  • Важко вносити зміни - Rigidity
  • Крихкість - Fragility
  • Неподвижність - Immobility
  • В'язкість - Viscosity

АБСТРАКЦІЯ

Не можливо закрити модуль від змін на всіх осях змін

Завжди будуть зміни, яких ми не очікували

Вибираємо напрямок, де ми очікуємо на зміни

Приймаємо першу кулю, а потім захищаємо себе від куль, що вилятають з тієї самої зброї

На практиці

  • Пишемо тести нашої системи
  • Розробляємо ітеративно
  • Часто показуємо фічі стейкхолдерам
  • Розробляємо спочатку найважливіші фічі
  • Часто робимо релізи

3. LSP - Liskov Substitution Principle

Підклас не повинен вимагати від коду, що його викликає, більше, ніж вимагав базовий клас. І повинен надавати цьому коду не менше, ніж надавав базовий клас.

Функції, які використовують посилання на базові класи повинні мати можливість використовувати об'єкти похідних класів не знаючи про це. 

			class Rectangle{
				set height(h){
				  this.height = h;
				}
				set width(w){
				  this.width = w;
				}
			}
		
			class Square extends Rectangle{
				set height(h){
					this.height = h;
					this.width = h;
				}
				set width(w){
					this.width = w;
					this.height = h;
				}
			}

if(object instanceof Square){}

Типові приклади порушення LSP

  • Дочірній клас робить менше, ніж базовий клас
  • Виключення в дочірніх класах
  • Функція, що має посилання посилання на базовий клас вимушена перевіряти який клас їй передали

4. ISP - Interface Segregation Principle

Клієнти не повинні залежати від методів, які вони не використовують

			interface Stream{
				void Read(){ }
				void Write(){ }
			}
			class File implements
							Stream{
				void Read(){ }
				void Write(){ }
			}
		
			class Keyboard implements Stream{
				void Read(){ }
				void Write(){
					throw new UnsupportedOperationException();
				}
			}
			class Printer implements Stream{
				void Read(){ 
					throw new UnsupportedOperationException();
				}
				void Write(){ }
			}

В чому проблема?

Жирний інтерфейс

			interface ReadableStream{
				void Read(){ }
			}
			interface WritableStream{
				void Write(){ }
			}
		
			class File implements ReadableStream, 
						WritableStream{
				void Read(){ }
				void Write(){ }
			}
			class Keyboard implements ReadableStream{
				void Read(){ }
			}
			class Printer implements WritableStream{
				void Write(){ }
			}

5. DIP - Dependency Inversion Principle

Application

FLOW of
CONTROL

Модулі вищого рівня не повинні залежати від модулей нижчого рівня. Всі вони повинні залежати від абстракцій.

Абстракція не повинна залежати від деталей реалізації. Деталі реалізації повинні залежати від абстракції.

Inversion of Control

Не має значення те, на якій мові написана програма.
